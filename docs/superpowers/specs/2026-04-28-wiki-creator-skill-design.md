**Status:** Approved (brainstorming complete, ready for implementation planning)
**References:**

## 1. Goal

Ship a Claude Code plugin called `wiki-creator` that bootstraps an LLM-maintained wiki — persistent agent memory — into any target project. Inspired by Karpathy's three-layer LLM Wiki and OmegaWiki's production-grade extensions, but generalized across domains and built around a strict mutability contract rather than a fixed directory layout.

The plugin contains **multiple skills**, not one. A meta skill (`wiki-init`) handles bootstrap; sibling skills handle ongoing operations (ingest, query, lint, evolve, spawn-agent, render). Output of a bootstrap is a working memory layer that subsequent agent sessions can read and extend without ad-hoc structure drift.

## 2. Non-goals

Out of scope for this plugin:

- **Embeddings / vector search.** Karpathy's argument: index.md is sufficient up to ~100 sources. If a project outgrows this, that is a separate concern.
- **Vector databases (pgvector, Milvus, OpenSearch).** Same reason.
- **Project-specific agents.** The plugin reserves slots and offers `wiki-spawn-agent`, but does not author domain-specific agents (KROS, Seldon, etc.).
- **Frontend / web UI for the wiki.** Obsidian and GitHub web view cover the human surface. Karpathy-style HTML dashboards may come later as a separate skill.
- **Redis or any runtime infrastructure.** This is a documentation-layer skill, not runtime plumbing.
- **Migration from existing documentation.** v2 concern, separate sub-skill.

## 3. Conceptual base

### Karpathy spirit (preserved)

- **Three layers, nothing more:** raw (immutable sources), wiki (LLM-generated markdown), schema (CLAUDE.md — runtime contract).
- **Three operations:** ingest, query, lint.
- **Two service files:** `index.md` (catalog) and `log.md` (append-only chronology).

### OmegaWiki extensions (adopted selectively)

- Typed entities with YAML frontmatter and required sections.
- Typed edge graph (`graph/edges.jsonl`): `extends`, `contradicts`, `supports`, `inspired_by`, `tested_by`, `invalidates`, `supersedes`, `addresses_gap`, `derived_from`.
- Bidirectional link enforcement — every forward link must have a reverse, formalized in CLAUDE.md.
- Auto-generated `graph/context_brief.md` (≤8000 chars compressed context) and `graph/open_questions.md`.
- Lint with `--fix` / `--suggest` / `--dry-run` modes.
- Skills as slash commands.
- GitHub Actions cron for periodic review.

### Additions beyond references

1. **Three-layer model expressed as mutability contracts**, not fixed folders. See §5.
2. **Domain presets** — software-project (default), research, product, personal, knowledge-base, custom.
3. **Architectural overlays** — none, clean, hexagonal, ddd, ddd+clean, ddd+hexagonal, layered. Optional, applied over a preset.
4. **Schema evolution** — wiki structure grows through ingest. New artifact types trigger an evolve flow that updates CLAUDE.md and creates frontmatter templates.
5. **Session-end automation** — every Claude Code session ends with a lint sweep, a log entry, and a context-brief regeneration.

## 4. Architecture decisions (from brainstorming)

| # | Decision | Rationale |
|---|---|---|
| 1 | Plugin shape: mono-plugin with 7 skills | Hybrid layered: meta skill bootstraps; ops skills are siblings; project-local skills can be generated by the meta when schema evolves. |
| 2 | Agents handled out-of-band | Bootstrap stays clean. Sibling skill `wiki-spawn-agent` adds agents on demand. Wiki = pure memory layer; agents = orthogonal concern. |
| 3 | Three layers = mutability contracts | Layers are not folders. They define who may write to which paths. CLAUDE.md publishes the matrix; lint enforces it. |
| 4 | UI = Obsidian-first | `.obsidian/` config always generated. HTML rendering is a sibling skill (`wiki-render`), not part of bootstrap. |
| 5 | Python 3.12 + uv | Modern, fast, isolated. `uv sync` creates `.venv`, no global pollution. |
| 6 | Default preset = software-project | Asked explicitly during interview, but software-project is the recommended default. |
| 7 | Wiki dir auto-detected | Greenfield → `wiki/`. Existing source detected (src/, lib/, package.json, Cargo.toml, etc.) → `.wiki/`. User can override. |
| 8 | Implementation approach: mono-plugin, Python+Jinja bootstrap | Deterministic, fast, testable. Templates are battle-tested. Shared `wiki_engine.py` avoids duplication across ops skills. |

§13 defaults from the predecessor doc are accepted as-is except where overridden by these decisions:

- #5 (skills location) → top-level skills in plugin; project-local skills generated only at schema-evolve time.
- #6 (Obsidian) → always generated, not asked.

## 5. The three-layer mutability model

Layers are not directories. They are write-permission contracts. CLAUDE.md publishes this matrix; lint enforces it.

| Layer | Path examples | Writer | Reader | Rule |
|---|---|---|---|---|
| **L1 Raw** | `raw/**` | Human only (manual drop, initial bootstrap) | Anyone | Immutable after ingest. Read-only for LLM. |
| **L2 Wiki** | `<wiki_dir>/**` excluding `<wiki_dir>/graph/**` | LLM via skills | Anyone | LLM-mutable. Each write triggers reverse-link validation. |
| **L3 Derived** | `<wiki_dir>/graph/**`, auto sections of `<wiki_dir>/index.md` | `tools/wiki_engine.py` only | Anyone | Auto-generated. Hand edits get clobbered on next rebuild. |
| **L4 Schema** | `CLAUDE.md`, `.claude/**`, frontmatter templates | Human + `/wiki-evolve` (gated) | Anyone | Contract layer. LLM mutates only via explicit evolve. |

Scaffolding outside the model: `tools/`, `.github/`, `tests/`, `pyproject.toml`. Plain plumbing, no special rules.

## 7. Bootstrap flow (`/wiki-init`)

```
1. ENVIRONMENT CHECK
   - Python ≥ 3.12 present
   - uv installed (else suggest `pipx install uv`)
   - git repo (else offer `git init`)
   - Detect existing source code in CWD

2. INTERVIEW (sequential, multiple-choice when possible)
   a. Project name (default: dir basename)
   b. One-line description
   c. Wiki dir: auto-suggest `wiki/` or `.wiki/` based on detection
   d. Preset: software-project | research | product | personal | knowledge-base | custom
   e. (if software-project) Architectural overlay — explicit choice, no default
   f. (if custom) Define entity types interactively
   g. i18n: EN-only (default) or EN + other
   h. Hooks installation: all | session-only | git-only | none
   i. CI workflows: yes (default if .github/ exists or remote is GitHub) | no
   j. Schema-evolve mode: gated (default) | auto

3. PROPOSE PLAN (gate)
   Show resolved tree + key decisions. User approves before any writes.

4. RENDER (scripts/bootstrap.py)
   - Resolve preset YAML + overlay YAML → merged config
   - Jinja-render every assets/*.j2 → target paths
   - Copy assets/hooks/, assets/workflows/, assets/obsidian/ as configured
   - Copy tools/ + pyproject.toml
   - Create empty entity-type dirs
   - Initialize <wiki_dir>/graph/edges.jsonl (empty)
   - Write first log.md entry: bootstrap

5. POST-RENDER
   - `uv sync` in target (creates .venv)
   - `tools/lint.py --dry-run` sanity check (must be clean)
   - `tools/wiki_engine.py rebuild-context-brief`
   - git add . && git commit -m "wiki bootstrap"

6. FIRST INGEST (optional)
   If raw/ has files, prompt to ingest now via /wiki-ingest.

7. HANDOFF
   - 3 main commands
   - How to open Obsidian on <wiki_dir>
   - session-end hook explainer
   - Suggest next: /wiki-spawn-agent if needed
```

### `InterviewConfig` schema

```python
@dataclass
class InterviewConfig:
    project_name: str
    project_description: str
    wiki_dir: str                              # "wiki" or ".wiki"
    preset: Literal["software-project", "research", "product", "personal", "knowledge-base", "custom"]
    overlay: Literal["none", "clean", "hexagonal", "ddd", "ddd+clean", "ddd+hexagonal", "layered"]
    custom_entity_types: list[EntityTypeSpec] | None
    i18n_languages: list[str]                  # ["en"] default
    hooks: Literal["all", "session", "git", "none"]
    ci: bool
    schema_evolve_mode: Literal["gated", "auto"]
    obsidian: bool = True
    python_version: str = "3.12"
    package_manager: str = "uv"
```

### Idempotency

Re-running `/wiki-init` in an existing wiki detects `CLAUDE.md` and enters upgrade mode: diff current structure against the requested preset+overlay, show changes, never touch existing wiki/ pages — only update schema and scaffolding when user confirms.

## 8. Op skills

| Skill | Trigger | Process | Outputs |
|---|---|---|---|
| `/wiki-ingest <paths…>` | User passes raw files | Detect type → classify → match slot or trigger evolve → generate page(s) → add edges → update index/log/graph | New `<wiki_dir>/<type>/<slug>.md`, edges, log entry |
| `/wiki-query <question>` | Free-text question | Read index.md → pick relevant pages → read them → synthesize → optionally save to `<wiki_dir>/outputs/<slug>.md` | Answer in chat, optionally new page |
| `/wiki-lint [--fix \| --suggest \| --dry-run]` | Manual or hook | Pure Python in `tools/lint.py`. No LLM. Deterministic | Lint report (🔴/🟡/🟢); `--fix` writes safe corrections |
| `/wiki-evolve <type-name>` | Manual or auto-invoked from ingest | Update CLAUDE.md (page types, frontmatter, cross-ref) → create dir → add frontmatter template → commit `[schema-change]` → append log | Schema diff + commit |
| `/wiki-spawn-agent <name> <role>` | Manual | Generate `.claude/agents/<name>.md`. Optionally companion skill in `.claude/skills/` | New agent definition |
| `/wiki-render <obsidian \| html>` | Manual | Obsidian: refresh `.obsidian/` config. HTML: render static site to `dist/` | Updated UI artifacts |

`/wiki-query` synthesizes only. New content enters via `/wiki-ingest`. Manual edits + `/wiki-lint` cover hand edits.

### Lint checks

| Check | Severity | Auto-fixable |
|---|---|---|
| Broken wikilink | 🔴 | No |
| Missing reverse link | 🟡 | Yes |
| Orphan page (no incoming + not in index) | 🟡 | No |
| Missing required frontmatter field | 🔴 | No |
| Duplicate slug or alias | 🔴 | No |
| Dependency rule violation (overlay-specific) | 🟡 | No |
| Contract `service` set but module doesn't list contract in `## Provides` | 🟡 | Yes |
| Contract version bumped without consumer migration note | 🟡 | No (suggests template) |

🔴 blocks pre-commit and CI. 🟡 warns, does not block.

### Schema-evolve flow

`/wiki-ingest` matching pseudocode:

```python
def ingest(path: Path) -> IngestResult:
    artifact = classify(path)
    slot = match_slot(artifact, schema_config)

    if slot is None:
        if config.schema_evolve_mode == "gated":
            options = propose_slot_options(artifact)   # (a) new top-level, (b) sub-folder, (c) section append
            choice = ask_user(options)
            if choice == "abort":
                return IngestResult.skipped()
            evolve_schema(choice)
            slot = choice.slot
        else:
            slot = best_default_slot(artifact)
            evolve_schema(slot, auto=True)

    page = render_page(artifact, slot)
    write(page)
    update_index(page)
    update_edges(page)
    append_log(f"ingest | {path.name} → {slot}")
    return IngestResult.created(page)
```

`/wiki-evolve` actions when invoked:

```
1. Update target's CLAUDE.md (page types table, frontmatter, cross-ref rules)
2. Create assets/frontmatter/<type>.yaml in target
3. mkdir <wiki_dir>/<dir>/
4. git commit -m "[schema-change] add type: <type>"
5. Append <wiki_dir>/log.md:
   ## [date] schema-change | added type: <type> (<path>) | trigger: <trigger>
```

### Project-local skill generation

When `/wiki-evolve <type> --generate-skill` is requested, the meta renders `assets/child-skills/ingest-domain.j2` into `target/.claude/skills/wiki-ingest-<type>/SKILL.md`. The child skill is a thin wrapper around `/wiki-ingest` with pre-classification baked in. It lives only in the target, never in the plugin.

### Cross-reference enforcement at write time

The `pre-tool-use` hook intercepts Write/Edit on `<wiki_dir>/**`:

- Validates frontmatter against the type's schema.
- If a forward link is added, calls `tools/wiki_engine.py add-edge --bidirectional` to write the reverse on the target page.
- If the target page is missing, creates a stub with TODO and logs `stub created, fill via /wiki-ingest`.

## 9. Hooks

### L1 — Claude Code session hooks

In target's `.claude/settings.local.json`, scripts in `.claude/hooks/`.

| Hook | Event | Action |
|---|---|---|
| `session-start.sh` | Session start | `cat <wiki_dir>/graph/context_brief.md` — loads compressed context for agent |
| `pre-tool-use` (Write/Edit on `<wiki_dir>/**`) | Before write | `tools/lint.py --pre-write <file>` validates frontmatter + reverse links |
| `post-tool-use` (Write on `<wiki_dir>/**`, exclude `graph/**`) | After write | Debounced (5s) `tools/wiki_engine.py rebuild-context-brief` |
| `session-end.sh` | Session end | 1) `lint --suggest` → save report. 2) Append session-end entry to `log.md`. 3) Echo summary: writes count + suggestions count. |

`session-end` does not auto-fix. User runs `/wiki-lint --fix` or `/wiki-review` manually if they want action on suggestions.

### L2 — Git hooks

Installed by `assets/install-hooks.sh` (run once during bootstrap if user opted in).

| Hook | Action |
|---|---|
| `pre-commit` | `tools/lint.py --fix` for safe corrections; **fail if 🔴 remain**. Stages auto-fixes. |
| `post-commit` | If `<wiki_dir>/**` touched → `wiki_engine.py rebuild-edges` regenerates `graph/edges.jsonl` from frontmatter. |

### L3 — CI (GitHub Actions)

| Workflow | Trigger | Action |
|---|---|---|
| `wiki-lint.yml` | push, pull_request | `uv run python tools/lint.py` — fail PR if 🔴; post 🟡 as PR comment |
| `wiki-review.yml` | weekly cron `0 9 * * 1` | Headless `claude -p "/wiki-review"` → publishes findings as GitHub issue tagged `wiki-review` |
| `wiki-rollup.yml` | monthly cron `0 9 1 * *` | Headless `claude -p "/wiki-rollup month"` → commits `<wiki_dir>/rollups/<YYYY-MM>.md` |

CI workflows require `ANTHROPIC_API_KEY` secret. Bootstrap does not configure secrets — README has a section "Enable CI workflows" with setup instructions.

## 10. Testing strategy

### Unit (`tests/unit/`)

Pure Python. Fast. No LLM. Tmpdirs for any file I/O.

| Module | Tests |
|---|---|
| `tools/wiki_engine.py` | Edge add/remove/query; deterministic context_brief output; open_questions extraction |
| `tools/lint.py` | Each check: positive case + each violation case; `--fix` mode produces correct output |
| `tools/classify.py` | Extension-based classification (no LLM for unambiguous cases); ambiguous case stub for LLM |
| `scripts/bootstrap.py` | Given InterviewConfig → produces expected file tree (snapshot test); idempotent re-run; upgrade mode diff |
| `scripts/interview.py` | Answers → InterviewConfig serialization round-trip |

### Integration (`tests/integration/`)

End-to-end bootstrap of sample projects. No LLM (LLM-dependent ops tested separately).

```
tests/integration/
├── fixtures/
│   ├── greenfield/                    # Empty dir → bootstrap → assert tree
│   ├── existing-codebase/             # Has src/, package.json → assert .wiki/ chosen
│   └── upgrade/                       # Existing wiki w/ old preset → assert upgrade
├── test_greenfield_software_hexagonal.py
├── test_greenfield_research.py
├── test_existing_codebase_autodetect.py
├── test_upgrade_preserves_wiki_pages.py
└── test_lint_clean_after_bootstrap.py  # Invariant: every bootstrap output passes lint clean
```

### Skill description tests (`tests/eval/`)

Skill-creator-style trigger phrase tests. 6-8 phrases per skill. Run via `claude -p` headlessly. Optional, not blocking initial release.

```
tests/eval/
├── trigger_init.yaml
├── trigger_ingest.yaml
├── …
└── anti_triggers.yaml
```

### Sample wiki fixture

`tests/fixtures/sample-wiki/` — tiny realistic wiki used by many tests:

- 3 modules with `depends_on` graph
- 2 decisions, 1 contract
- One intentional orphan + one intentional missing-reverse for negative tests

### Infrastructure

- Runner: `pytest`
- `uv sync --dev` installs pytest + plugins
- Plugin's own GitHub Actions runs `uv run pytest` on push
- Coverage target: 80%+ on `tools/` and `scripts/`

### Out-of-test (manual)

- `/wiki-query` answer quality — judged subjectively
- `/wiki-review` suggestion quality — same
- Obsidian rendering — eyeball after bootstrap

## 11. Trigger phrases

Skill description for `wiki-init`:

> Use this skill whenever the user wants to bootstrap a wiki, knowledge base, or persistent memory layer for an LLM agent or multi-agent system. Triggers include: "create a wiki", "set up a knowledge base", "bootstrap project memory", "agent memory layer", "Karpathy LLM wiki", "OmegaWiki style", "Obsidian + Claude Code workflow", "set up CLAUDE.md", "wiki for my project", "memory for agents". Also use when the user describes wanting raw → wiki → schema separation, persistent compounding knowledge instead of RAG, or session-end automation that keeps documentation current. Use even if the user does not literally say "wiki" but describes a structured, LLM-maintained knowledge base with cross-references and lifecycle. Do NOT use for one-off doc generation, single-file summaries, or static documentation that won't be incrementally maintained.

Each ops skill gets its own description focused on its narrower trigger surface. Eval suite (§10) tunes them.

## 12. Sizing

- `SKILL.md` for each skill: 200-450 lines (under skill-creator's 500 limit)
- `references/`: 8-10 files, 100-300 lines each
- `assets/`: ~25-35 files (templates, hooks, workflows, child-skills)
- `scripts/`: 3 Python files, 100-300 lines each
- Eval test suite: 6-8 prompts per skill

Bootstrap empty project: a few seconds. First ingest of 5-10 documents: a few minutes.

## 13. Open questions deferred

These are flagged for the implementation plan, not blocking the spec:

- Headless Claude Code invocation in CI — exact incantation depends on Claude Code CLI version available in GitHub Actions.
- Obsidian Dataview vs static index.md — current decision is materialized index.md (LLM reads raw markdown, not Dataview output). May revisit if Obsidian-only workflows surface.
- `tools/classify.py` LLM fallback — implementation can start with extension-only classification; LLM call added later if test cases prove it necessary.
- `wiki-render` HTML output template — minimal viable static site, refine after first dogfooding.
